## 1. 회원가입 구조 (센터 기반 통제 시스템)

### 핵심 개념
회원가입은 단순 가입이 아니라 **"센터 단위 통제 + 셀러 데이터 축적" 구조**

### 1.1 1차 필수 입력 정보 (무조건 입력)
- 아이디
- 비밀번호
- 이름
- 휴대폰번호
- 소속관리자 이름
- **센터명 (필수)** ← 여기까지 입력 안 하면 가입 불가

### 1.2 소속관리자 / 센터코드 구조 (핵심)

**이 시스템은 센터 기반으로 운영**되기 때문에 모든 계정은 반드시 센터에 소속되어야 합니다.

소속된 센터별로, 셀러들의 매출이나 발주내역은 전부 볼 수 있습니다.

#### 센터코드 규칙
```
01 = 서울
02 = 경기
03 = 인천
04 = 부산
05 = 대구
06 = 대전
07 = 광주
08 = 울산
09 = 세종
10 = 강원
11 = 충북
12 = 충남
13 = 전북
14 = 전남
15 = 경북
16 = 경남
17 = 제주
```

#### 센터코드 조합
센터코드 + 센터 대표자 폰뒷 번호 조합으로 구성

**예시**:
- 01-4213 (서울)
- 02-3413 (경기)
- 14-6521 (전남)

#### 중요 규칙
- 소속관리자(센터코드)가 없으면 방송 불가
- 모든 권한은 센터 기준으로 묶임

### 1.3 2차 추가 정보 (셀러 데이터용)
가입 이후 추가로 입력:
- 방송 채널 (그립 / 클릭메이트 / 기타)
- 월 평균 방송 매출
- 주요 판매 카테고리
- 활동 지역
- 방송 가능 시간대

### 1.4 센터 가입
센터에서 가입하려고 하면 별도의 센터가입 관련 계약서 및 아이디와 계정을 받아야 합니다.

필수 정보:
- 주소
- 대표자
- 사업자등록증 등 세부 정보

여기에서 별도로 센터들은 가입하는 구조로 가고 코드를 부여받습니다.

## 2. 라이브커머스 바코드 운영 시스템

### 2.1 목적
이 프로그램은 단순히 재고를 보는 시스템이 아니라, **라이브 방송 중 '판매 판단'을 빠르게 하기 위한 도구**입니다.

### 현재 문제점
- 상품 보고 검색 따로 하고
- 가격 계산 따로 하고
- 멘트 고민 따로 하는 구조

→ 이렇게 하면 방송 흐름이 끊기고, 결국 판매 타이밍을 놓치게 됩니다.

### 핵심 목적
**바코드 한 번 찍으면, 판매에 필요한 모든 정보가 바로 나오게 만든다**

### 2.2 기본 동작 흐름

```
1. 바코드를 찍는다
2. WMS에서 재고와 상품 정보를 가져온다
3. 동시에 외부 시세를 자동으로 조회한다
4. 가격 / 마진 / 경쟁력 판단이 바로 보인다
5. AI가 판매 방법까지 제안한다
6. 바로 주문을 넣거나 방송을 진행한다
```

**이 전체 과정이 최대한 빨리 끝나야 합니다.**

### 2.3 네이버쇼핑 / 쿠팡 시세 자동 조회 기능

#### 현재 방식의 문제
버튼을 눌러서 검색하는 방식 → 방송에서는 거의 못 씁니다.

#### 변경 방향
**바코드를 찍으면 나오는 상품명을 기준으로**:
- 네이버쇼핑 + 쿠팡을 자동으로 검색
- 결과를 바로 띄움

#### 중요한 점
"검색 버튼"이 아니라 **기본적으로 항상 데이터가 떠 있어야** 합니다.

#### 표시 정보
- 네이버 최저가
- 쿠팡 최저가
- 평균 가격
- 리뷰 많은 상품 기준 가격

그리고 우리 판매가랑 비교해서:
- 우리가 비싼지
- 우리가 싼지
- 경쟁력이 있는지

**이걸 바로 판단할 수 있게 만들어야 합니다.**

#### 속도 최적화
같은 상품을 여러 번 찍을 수 있기 때문에:
- 한 번 조회한 데이터는 일정 시간 **캐싱**
- 빠르게 보여줘야 함

### 2.4 가격 / 마진 / 판매가 관리 기능

#### 핵심
이 부분은 단순 표시가 아니라 **"WMS 기준 통제 시스템"**으로 만들어야 합니다.

#### 문제점
화면에서 가격을 수정하거나 단순히 숫자만 보여주는 구조로 가면:
- 셀러마다 가격이 무너지고
- 결국 마진이 깨지는 상황 발생

#### 올바른 구조
**모든 가격 기준은 WMS에서 관리**:
- 공급가
- 최소 판매가
- 권장 판매가
- 최대 판매가

→ 이 값들은 **WMS를 수정하지 않으면 절대 변경되지 않아야** 합니다.

#### 시스템에서의 역할
현재 바코드 화면이나 시스템에서는:
- 가격은 **조회만 가능**
- **수정은 불가능**

#### 가격 변경 절차
```
MD → 내부 협의 → WMS 수정 → 시스템 반영
```
이 순서로만 변경되도록 해야 합니다.

### 2.5 AI 분석 기능 (가장 중요한 부분 중 하나)

#### 핵심
이 기능은 단순 정보 제공이 아니라 **"셀러를 대신해서 판매를 만들어주는 기능"**입니다.

#### 배경
초보 셀러들은 상품을 보고도 어떻게 팔아야 할지 모릅니다.

#### AI가 제공해야 하는 내용
- 이 상품의 핵심 포인트
- 누구에게 팔아야 하는지
- 방송에서 어떻게 말해야 하는지
- 같이 팔면 좋은 상품
- 주의해야 할 부분

#### 중요한 점
AI 답변은 **항상 일정한 구조**로 나와야 셀러들이 바로 사용할 수 있습니다.

#### 동작 방식
**바코드 찍으면 자동으로 실행** ← 이게 기본입니다.

#### 속도 문제 해결
- 이미 분석된 상품은 **저장**
- 필요할 때 다시 생성할 수 있게

### 2.6 네이버 시세 자동 표시 (기본값 설정)

#### 현재 문제
버튼 눌러서 보는 구조

#### 변경 방향
- 화면에 들어오면 **자동으로 시세가 떠 있어야** 함
- 상품이 바뀌면 **자동으로 갱신**

#### 중요한 기능
단순 가격 표시가 아니라 **"판단"**이 가능해야 합니다.

예를 들어:
- 우리가 평균보다 20% 비싸면 → **경고 표시**
- 우리가 최저가면 → **강조 표시**

이런 식으로 **셀러가 생각하지 않아도 바로 판단**할 수 있게 만들어야 합니다.

### 2.7 주문 접수 기능 (방송용 필수 기능)

#### 핵심
이건 따로 떼서 보면 안 되고 **방송 흐름 안에서 바로 입력 가능**해야 합니다.

#### 구조 (최대한 단순)
```
1. 수량 입력
2. 버튼 클릭
3. 자동 금액 계산
4. 나중에 엑셀 다운로드 가능하도록
```

**엑셀 양식은 WMS 발주 양식하고 동일**하게 하여 발주 시, 바로 입력될 수 있게

### 2.8 UI 구성 방향

#### 핵심
이건 디자인이 아니라 **속도와 판단 중심**으로 설계해야 합니다.

#### 구조
```
상단: 상품명, 재고, 가격/마진
중앙: AI 분석, 시세 정보
하단: 주문 입력, 방송 시작 버튼
```

#### 색상 의미
- 재고 많음 → **초록**
- 부족 → **노랑**
- 위험 → **빨강**

→ **한눈에 상태가 보이게**

## 3. 발주현황 관리 기능 (WMS 연동 자동화 구조)

### 핵심
이 부분은 단순 발주 관리가 아니라 **"셀러 → 발주 → 입금 → WMS → 출고"까지 자동으로 이어지는 구조**

### 3.1 현재 문제점
수동으로 발주 확인하고 처리하는 방식으로 가면:
- 속도가 안 나오고
- 인력 계속 붙어야 합니다

### 3.2 전체 구조

#### 발주서 업로드 자동 인식
셀러들이 각 플랫폼(클릭메이트, 그립 등)에서 사용하는 **발주서 양식이 다르기** 때문에:

발주서를 업로드하면 **자동으로 데이터를 인식**해서 → WMS에 바로 발주가 들어갈 수 있도록 만들어야 합니다.

#### 입금 여부 기준

**중요한 기준**: **입금 여부**

```
셀러가 발주서를 업로드
→ 바로 재고 차감이 아니라
→ "판매 권한만 선점"
```

#### 예시
WMS 재고가 100개인데 A 셀러가 발주서로 30개를 올리면:
- 실제 차감은 안 함
- A 셀러는 30개에 대한 **판매 권한을 먼저 가져감**

#### 입금 완료 시
```
입금 완료
→ WMS에 자동으로 발주 입력
→ 바로 출고 진행
```

#### 입금 안 된 상태
```
재고는 유지
다른 셀러는 나머지 수량만 판매 가능
```

### 3.3 자동 해제 조건

발주 후 **3시간 내 입금이 없으면**:
```
→ 해당 발주는 자동 취소
→ 판매 권한 해제
→ 전체 재고 다시 오픈
```

셀러가 직접 발주 취소해도 **동일하게 초기화**

### 3.4 향후 방향

#### 가상계좌 자동 발급
```
셀러들이 발주를 하면
→ 가상계좌 자동 발급

그 계좌로 입금이 들어오면
→ 자동 입금 확인
→ 자동 발주 생성 (WMS)
→ 자동 출고 진행
```

→ **사람이 개입 안 하는 구조**로 가야 합니다.

### 3.5 확장 기능: 셀러-상품 매칭 데이터 축적

셀러들이 발주한 데이터를 기반으로:
- 어떤 셀러가 어떤 상품을 잘 파는지
- 카테고리별 강점
- 판매 성과

**이걸 계속 쌓아야 합니다.**

#### 자동 추천 구조
이 데이터를 기반으로 **상품 소싱 시 자동 추천**:

```
건강기능식품 → 월 5회 이하
잡화 → 월 5회 이하
의류 → 월 10회 이하
```

이런 기준을 잡고 → 셀러별 / 상품별 조건에 맞는 → **가장 잘 팔릴 가능성이 높은 매칭을 자동으로 띄워주는 구조**

### 정리
- 발주는 **"입금 기준 자동화"**
- 재고는 **"선점 구조"**
- 출고는 **"WMS 자동 연동"**
- 데이터는 **"셀러 매칭용으로 축적"**

## 4. 상품관리 구조 (본사상품 + 센터상품)

### 4.1 왜 이렇게 나누는지 (핵심 이유)

#### 현실
전국 모든 센터의 상품을 100% 우리 물류창고(WMS)에 넣는 건 **불가능**합니다.

예를 들어:
- 각 센터마다 가지고 있는 상품이 다르고
- 물량도 다르고
- 지역도 다르고
- 입고/출고 방식도 다릅니다

→ 이걸 전부 우리 창고로 보내서 관리하는 건 **시간 / 비용 / 효율 다 맞지 않습니다**.

#### 결론
센터가 가진 상품은 **센터가 직접 관리하는 구조**로 가야 합니다.

### 4.2 왜 시스템에 입력하게 하냐 (중요)

#### 문제점
센터들이 자기 상품을:
- 따로 엑셀로 관리하고
- 따로 재고 보고
- 따로 방송하면

→ **이 시스템을 안 씁니다.**

#### 해결책
```
센터 상품도 시스템에 넣게 만들고
→ 바코드를 찍으면
→ 그 상품 정보가 바로 나오게
```

### 4.3 셀러 입장에서 변화

#### 기존 방식
```
상품 따로 확인
재고 따로 확인
가격 따로 계산
```
→ 복잡하고 느림

#### 시스템 사용 시
```
바코드 찍으면
→ 재고 보이고
→ 가격 보이고
→ 바로 방송 가능
```

→ 그래서 자연스럽게 **이 시스템으로 방송하게 되는 구조**

### 4.4 센터가 해야 하는 역할

센터는 본인 상품에 대해:
- 상품 등록
- 재고 입력
- 수량 수정

→ **이걸 직접 관리**해야 합니다.

#### 특히 중요한 것
**방송 전에 재고 업데이트**

이걸 안 하면:
```
과판매 → 배송 지연 → 클레임 발생
```

### 4.5 발주 분리 구조 (본사 vs 센터 자동 분기)

#### 핵심 개념
상품은 하나의 발주서로 들어오지만 → **출고 주체에 따라 자동으로 나눠야** 합니다.

#### 기준
```
본사 상품 → 우리 WMS 기준 → 자동출고
센터 상품 → 우리 WMS에 없음 → 센터출고
```
→ 이 기준으로 **발주를 자동 분리**

#### 자동 처리 결과
하나의 발주서가 → 아래 2개로 자동 분리됩니다:

1. **슈퍼무진 발주서 (본사)**
   - WMS에 있는 상품만 포함
   - 자동으로 WMS 발주 연동
   - 자동 출고 진행

2. **자사몰 발주서 (센터)**
   - WMS에 없는 상품만 포함
   - 센터별로 전달
   - 센터가 직접 출고

→ 최종적으로 **2개의 엑셀 시트로 자동 생성**:
- `슈퍼무진 발주서.xlsx`
- `자사몰 발주서.xlsx`

### 4.6 출고 방식

#### 고객 입장
2건 배송:
- 본사 상품 → 본사 택배
- 센터 상품 → 센터 택배

→ 시스템에서 **자동 분리되어 처리**

### 4.7 센터 상품 관리 방식 (중요)

센터 상품은 WMS가 없기 때문에 → **"센터가 직접 관리"**해야 합니다.

#### 센터 필수 기능
- 상품 등록
- 재고 수정
- 초기화
- 엑셀 업로드

→ **매일 방송 전** 센터는 반드시 **본인 상품 재고 업데이트**

### 정리 (한 줄 핵심)
→ 모든 상품을 우리 창고로 모을 수 없기 때문에
→ 센터 상품은 각자 관리하되
→ **시스템 안에서 같이 운영되도록 만든 구조**

## 5. 방송 캘린더 / 방송 운영 구조

### 5.1 기본 개념
방송 캘린더는 단순 일정 관리가 아니라 → **센터가 방송을 만들고 셀러를 움직이는 운영 구조**

### 5.2 방송 생성 구조
```
센터 로그인
→ 방송 일정 생성
→ 소속 셀러에게 배정
```
→ **센터 = 기획 / 셀러 = 실행**

### 5.3 셀러 이동 구조 (확장 핵심)

- 셀러는 **소속 제한 없음**
- **다른 지역 센터 방송 참여 가능**

→ **전국 단위 방송 풀 확장 가능 구조**

### 5.4 방송 시작 방식 (센터코드)

방송 시작할 때, 방송시작 누르고, **그 센터의 해당 코드를 입력**해야 센터에 입고되어진 제품리스트를 공유받을 수 있습니다.

```
셀러가 방송 시작 시
→ 센터코드 입력
→ 해당 센터 방송 연결

연결 즉시
→ 해당 센터 상품 리스트 자동 로드
```

### 5.5 방송 중 상품 사용

센터가 사전 등록한 상품 사용 (본사 + 센터 상품 포함)

→ 셀러는 **상품 찾지 않고 바로 바코드로 판매**

### 5.6 방송 종료

```
방송 종료 시
→ 연결 자동 해제
→ 상품 리스트 초기화
→ 데이터 저장 완료
```

다음 방송은 **다시 센터코드 입력**

### 5.7 데이터 축적 구조

모든 방송 데이터는 아래 기준으로 저장:
- 셀러별 매출
- 센터별 매출
- 방송별 성과

→ 이를 통해:
- 매출 잘 나오는 센터
- 센터별 강점 셀러

→ **자동 분석 가능**

### 5.8 핵심 구조 정리

- 센터는 방송과 상품을 제공
- 셀러는 어디든 가서 방송 실행
- **센터는 플랫폼, 셀러는 실행자**

### 한 줄 핵심
→ **센터가 판을 만들고, 셀러가 그 판에서 판다**

## 6. 상품제안 (샘플 발주 시스템)

### 6.1 목적

**하나의 "내부 전용 샘플 쇼핑몰" 구축**

```
센터가 제품 선택
→ 샘플 장바구니
→ 일괄 발주
→ 출고까지 자동화
```

### 6.2 핵심 개념 (이 페이지의 본질)

이건 쇼핑몰이 아니라 → **"샘플 발주 인터페이스"**

- 구매 목적 ❌
- 테스트/방송 준비 목적 ⭕

### 6.3 페이지 구성 구조

#### 3-1. 메인 (상품 리스트 페이지)

**구성**:
- 카테고리 분류
  - 식품 / 건기식 / 의류 / 잡화 / 유아 / 애견 / 기타
- 상품 카드 UI
  - 썸네일
  - 제품명
  - 공급가
  - 샘플가 (무료/유료 표시)
  - 재고 상태 (표시용)
  - 샘플 가능 여부

**기능**:
- 검색
- 카테고리 필터
- 정렬 (인기 / 재고 / 신규)

#### 3-2. 상품 상세 페이지

**구성**:
- 상세 이미지 / 설명
- 옵션 선택
- 샘플 정책 표시
  - 무료 / 유료 조건

**기능**:
→ "샘플 요청하기"

#### 3-3. 샘플 장바구니 구조

**일반 쇼핑몰 장바구니와 동일한 UX**

**기능**:
- 여러 상품 담기
- 수량 기본 1 (수정 제한 가능)
- 상품별 샘플가 합계 표시

→ 핵심 버튼: **"샘플 일괄 주문하기"**

#### 3-4. 샘플 일괄 발주

**동작**:
```
장바구니 → 일괄 발주 생성
```

**발주 데이터**:
- 발주유형: 샘플
- 상태: 입금대기
- 상품 리스트 (다건)
- 총 금액

#### 3-5. 결제 및 발주 흐름

```
발주 생성
→ 입금 안내 문자 자동 발송
→ 입금 확인
→ 발주 확정 처리
→ WMS 출고 요청
```

### 6.4 재고 관리 방식

**재고는 WMS 기준으로만 관리**

**방식**:
- 시스템에서는 재고 **"표시만"**
- 실제 차감은 **WMS 출고 시 자동 차감**

### 6.5 샘플 남용 방지 (통제 로직)

- 동일 상품 **5회 제한**
- **센터별 요청 수 제한**

### 6.6 전체 흐름 요약

```
1. 상품 리스트 확인
2. 상품 클릭
3. 샘플 요청 → 장바구니 담기
4. 샘플 일괄 주문
5. 입금
6. 발주 확정
7. WMS 출고
8. 샘플 수령
9. 방송 진행
```

### 핵심 정리
이 페이지는:
- "쇼핑몰"이 아니라
- **"센터용 샘플 발주 시스템"**

- UI는 쇼핑몰처럼
- 구조는 발주 시스템
- 재고는 WMS 기준
- 장바구니로 일괄 처리
- 결제 → 출고 자동 연결
